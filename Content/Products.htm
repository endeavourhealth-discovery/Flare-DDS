API

DDS APIs allow third party access to the DDS.

For example:

  • Authenticate - a valid authentication token for calling subsequent APIs.
  • Get FHIR resource types - a list of FHIR resource types, and associated (human readable) names, that the DDS currently recognises, stores, and supports.
    See also:
    FHIR profile resources
    GitHub FHIR Profiles repository
  • Get patients for NHS number - summaries for all patients known to Discovery with the given NHS number.
  • Get resources for patients - FHIR resources of the requested type(s), for the requested patient(s).
  • Get referenced resource - details of the requested resource reference such as doctors, locations, or organisations.
  • Get flag for patient - the current state of the requested flag (for example frailty) for the requested patient.

    • Important: The APIs and algorithms will only use data, return resources, or identify patients that you have protocols, agreements, and access for.

See Discovery APIs for more details.

If you want to test our APIs against your own software, register your interest.

Extract

A priority for those who support healthcare is the ability to implement data-driven quality improvements. The power of the DDS is to provide a single source of data from multiple organisations, enabling you to analyse linked primary, acute, urgent care and community data. The Data Extract tool provides access to defined populations and data sets to enable local organisations to interrogate the data using their preferred analytical tools.

Query

Achieving personalised predictive analytical capability depends upon access to real time clinical data, collected at the point of care. The DDS contains data, from across primary, secondary, urgent care, and community clinics, facilitating access (under strict IG rules) to an up to date logical equivalent of patient and related data. The Query Engine is designed to ensure we maximise the benefits of data held in the DDS by making that data available to end users. It is designed to support:

  • advanced cohort query and large-scale output reports with a view to more sophisticated visualisation techniques (dash boarding) of reports.
  • a number of secondary uses, in particular where there is a need to analyse linked primary and acute data.
  • better access by publishers to their own data for direct care purposes.
Referencing Data Sharing Agreements and defined cohorts and populations, data may be made available in identifiable; de-identified or anonymised states, depending on the purpose. For example, direct care requires access to identifiable, whereas statistical reporting may only require anonymised data
