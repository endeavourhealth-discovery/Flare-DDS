Information Model – language and structural specification

Last update: 16 January 2020

Introduction

Objectives of the information model

The objectives of the information model are to:

1. Enable users to visualise and understand the structure and content of health records. The users should not have to be technical experts or informaticians.

2. Provide a specification of structure and content for systems that store or query health data.

3. To create a single unified language (subset of UML and description logic) that is both human readable and machine interpretable to represent the structure, content and query.

4. To provide an ontology (or set of ontologies) which can be used by reasoners and subsumption query to support advanced analytics.

5. To provide a means of specifying the transforms or mappings between data sets or to define reports that extract data from data sets.

It is possible to envisage health record entries as an unlimited set of concepts each with an unlimited set of properties or relationships, each with an unlimited set of values or relationship targets.

Such a mechanism would be difficult to make sense of. For example, dates may be represented in thousands of different ways and it may not be clear what a particular date refers to. Visualisation of patterns, such as that used by the Internet search engines, would not meet requirements.

The information model seeks to make sense of the plethora of data points by categorising attributes and values, and organising them in a way that meets people’s perception of the information contained in the record stores.

In addition, the information model provides machine interpretable instructions to implementors when designing data bases or data sets.

What is an information model?

A model is a representation of something real.

An information model is a model of information. Information is a description of an arrangement of data for particular purposes.

More specifically, in this context, an information model is tailored to illustrate arrangements of health and care data as held in health records for the purposes of understanding and query.

In order to build a system there is a need for an architectural designs. In software terms an architectural design is an “abstract model”. The information model structure can thus be considered an abstract model.

The effect of the information model is to form a bridge between the human conception of health care records and the machine-based implementation of those records so that data within the records can be accessed and used more easily and efficiently by people who need information.

In order to bridge the conceptual gap, the model creates a series of machine interpretable conceptualisations, each of which gets closer to an underlying machine readable logical information model, which itself then maps directly to implementation schemas and query languages, whether they be relational, graph or other.

The model uses a layered approach.
